#### Plan Differences
Plan size: A=36 nodes / 36 edges, B=53 nodes / 46 edges (+17 nodes, +10 edges).

Operator-by-operator delta (from `compare sql --plans`):

- Scans: 4 `CometNativeScan` (date_dim×2, store, store_returns, store_sales) → 4 `FileSourceScanExecTransformer` for the same tables. Symmetric.
- Broadcast side: 3 `CometBroadcastExchange` + 3 `CometBroadcastHashJoin` + 1 `CometSubqueryBroadcast` → 3 `ColumnarBroadcastExchange` + 3 `BroadcastHashJoinExecTransformer` + 1 `ColumnarSubqueryBroadcast`. Symmetric.
- Probe-side join (the one large join): **1 `CometSortMergeJoin` + 2 `CometSort` + 3 `CometExchange` (Comet)** vs **1 `ShuffledHashJoinExecTransformer` + 3 `ColumnarExchange` (Gluten)**. Comet performs a sort-merge join with explicit pre-sorts on both probe inputs; Gluten performs a shuffled hash join with no sort.
- Aggregation: 2 `CometHashAggregate` (partial+final) → 1 `FlushableHashAggregateExecTransformer` + 1 `RegularHashAggregateExecTransformer`. Functionally equivalent.
- Sink: 1 `CometTakeOrderedAndProject` → 1 `TakeOrderedAndProjectExecTransformer`. Symmetric.
- Gluten plan additionally contains 7 `WholeStageCodegenTransformer` wrappers, 6 `InputIteratorTransformer`, 3 `VeloxResizeBatches`, 5 `FilterExecTransformer`, 8 `ProjectExecTransformer`, 1 `VeloxColumnarToRow` (vs 1 `CometNativeColumnarToRow` in Comet). These account for the +17 node count without changing data flow.

Net structural difference: **Comet uses sort-merge join + 2 explicit sorts; Gluten uses shuffled hash join with no sort.**

#### Stage Analysis

**Build/scan stages (symmetric):**
- Comet 4246 vs Gluten 11955 (store_sales scan + shuffle write, 3193 tasks): A=22.211s, B=21.591s, Δ=-620ms. Input 52.6→54.2 GB (+1.5 GB), shuffle write 118.4→120.9 GB (+2.5 GB). Task p50 duration 752ms→796ms (+44ms). Essentially equivalent — not the bottleneck.
- Comet 4247 vs Gluten 11956 (store_returns scan + shuffle write, 111 tasks): A=21.221s, B=21.284s, Δ=+63ms. Equivalent.

**Join/probe stage (the bottleneck):**
- Comet 4250 vs Gluten 11959 (200 tasks each): **A=1m22.2s, B=9.764s, Δ=-1m12.436s.**
  - This single stage accounts for nearly the entire query-level delta of -1m13.025s.
  - Task duration p25/p50/p75/max: A=38.545s/40.306s/42.823s/46.025s, B=3.956s/4.967s/5.526s/6.814s. Per-task p50 delta = -35.339s (~8.1× faster on Gluten).
  - Shuffle Read aggregate: A=23.9 GB, B=121.0 GB. Gluten reads ~5× more bytes per task (p50 shuffle read B=619.7 MB vs A=0 B at p50; A's total 23.9 GB is concentrated in a small subset of tasks — A reports max=634.5 MB but p75=0 B, while B reports a uniform ~620 MB across all 200 tasks). The Comet write→read asymmetry (118.6 GB written → 24.0 GB read on the same data) appears to be a metric-reporting artifact of `CometShuffleManager`; the upstream stages produced the same record count (7,959,519,248 records) on both engines.
  - Peak Exec Memory: A=0 B for all quantiles, B=184 MB p25/p50/p75 / 192 MB max. Comet does not surface native peak memory through this metric, so this is not directly comparable, but Gluten's hash table fits in ~184 MB per task.
  - Shuffle Write (downstream partial-agg output): A=48.7 MB, B=29.7 MB. Both write ~135,000 records out of the join.
  - GC Time: A=30ms, B=10ms. Δ=-20ms. Negligible.

**Trailing stages** (final aggregate + take-ordered): Comet 4254 = 253ms, Gluten 11963 = 138ms; Comet 4255 = 36ms, Gluten 11964 = 15ms. Sub-second, not material.

#### Root Cause
The +69% Gluten Velox lead is concentrated in a single stage: the probe-side join over the (store_sales × store_returns) shuffled outputs. Comet's plan emits `CometSortMergeJoin` with two preceding `CometSort` operators on the shuffled inputs (200 partitions, ~135M joined records), while Gluten emits `ShuffledHashJoinExecTransformer` with no sort. Comparing matched stages, Comet 4250 runs 1m22.2s while Gluten 11959 runs 9.764s — a -1m12.436s delta that explains essentially all of the -1m13.025s SQL-level delta. Per-task p50 duration drops from 40.306s to 4.967s (8.1× faster) even though Gluten reports 5× more shuffle-read bytes per task (619.7 MB vs Comet's heavily skewed 0 B p50 / 634.5 MB max).

Upstream scan + shuffle-write stages are within 1s of each other (22.211s vs 21.591s for the 3193-task store_sales scan; 21.221s vs 21.284s for store_returns), so the input pipeline, broadcast joins, and shuffle producers are not the source of the gap. Spill is zero on both sides. GC time is small on both (120ms vs 15ms total). The +6791 task count on Gluten is driven by the additional `WholeStageCodegenTransformer`/`InputIteratorTransformer`/`VeloxResizeBatches` wrapper nodes accruing tasks across the same 200-partition shuffles, not by additional data movement.

Net root cause: **Comet selected a sort-merge join (with two explicit sorts) for the large probe-side join while Gluten selected a shuffled hash join.** The sort-merge path forced ~40s of per-task sort+merge work in stage 4250; the shuffled-hash path completes the same join in ~5s per task. Every other component of the query (scans, broadcast joins, partial aggregates, top-N) is within noise between the two engines.
