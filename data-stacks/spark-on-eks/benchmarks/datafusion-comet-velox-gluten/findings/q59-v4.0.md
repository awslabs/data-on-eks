#### Plan Differences
Same 16 stages and matching 3193-task scan partitioning on both sides; node count rises from 73 (Comet) to 108 (Gluten) primarily from explicit `WholeStageCodegenTransformer` (15) and `InputIteratorTransformer` (12) wrappers Velox emits around each pipeline boundary.

Operator-class swaps (1:1 mapping, same join graph):
- Scans: `CometNativeScan` store_sales×2, store×2, date_dim×6 → `FileSourceScanExecTransformer` (same counts).
- Filters/Projects/Aggregates: `CometFilter`×10 / `CometProject`×11 / `CometHashAggregate`×8 → `FilterExecTransformer`×10 / `ProjectExecTransformer`×15 / `FlushableHashAggregateExecTransformer`×4 + `RegularHashAggregateExecTransformer`×4.
- Broadcast joins: `CometBroadcastHashJoin`×6 + `CometBroadcastExchange`×6 → `BroadcastHashJoinExecTransformer`×6 + `ColumnarBroadcastExchange`×6.
- Final join: `CometSortMergeJoin`×1 (with `CometSort`×2 + `CometExchange`×6) → `ShuffledHashJoinExecTransformer`×1 (with `ColumnarExchange`×6, no sort). Comet picked SMJ; Gluten picked SHJ.
- Row conversion: `CometNativeColumnarToRow`×3 → `VeloxColumnarToRow`×3.
- Top-N: `CometTakeOrderedAndProject`×1 → `TakeOrderedAndProjectExecTransformer`×1.
- Gluten-only: `VeloxResizeBatches`×6 between exchanges and operators.

#### Stage Analysis
Both runs are dominated by two store_sales scan stages (3193 tasks each) feeding the two CTEs of the query.

Comet store_sales scans:
- Stage 4432: 20.297s wall, 18.9 GB input, 8,251,110,748 records, p25/p50/p75/max task = 259ms / 299ms / 366ms / 1.592s, shuffle write 88.1 MB.
- Stage 4431: 11.55s wall, 18.9 GB input, 8,251,110,748 records, p25/p50/p75/max task = 291ms / 344ms / 428ms / 1.646s, shuffle write 87.6 MB.

Gluten store_sales scans:
- Stage 4415: 7.997s wall, 4.1 GB input, 1,685,900,972 records, p25/p50/p75/max task = 35ms / 40ms / 75ms / 820ms, shuffle write 52.1 MB.
- Stage 4414: 4.57s wall, 4.1 GB input, 1,687,079,402 records, p25/p50/p75/max task = 37ms / 56ms / 146ms / 865ms, shuffle write 52.5 MB.

Per-stage deltas at the two scan stages (matching pairs):
- Bytes read per stage: 18.9 GB → 4.1 GB (≈4.6× less in Gluten).
- Records read per stage: 8.25B → 1.69B (≈4.9× fewer in Gluten).
- Median task time: 299/344 ms → 40/56 ms (≈6–7× faster).
- Stage wall time: 20.297s + 11.55s = 31.85s → 7.997s + 4.57s = 12.57s (-19.28s of the -12.36s end-to-end gap; remaining stages run in parallel with these).
- Total stage time (executor-CPU sum, across all 16 stages): 40.925s → 16.626s (-24.299s).

Shuffle Read at the SQL level (175.7 MB → 147.4 MB, -28.3 MB) and Shuffle Write (201.1 MB → 147.4 MB, -53.6 MB) are nearly identical, confirming both engines move similar post-aggregation row volumes; the divergence sits entirely on the scan side. GC was a non-factor on either run (167ms vs 40ms, no spill).

#### Root Cause
The 12.36s gap is concentrated in the two store_sales scan stages, which together account for 31.85s of stage time on Comet vs 12.57s on Gluten Velox. Both plans run the same 16-stage shape with identical 3193-way partitioning on store_sales, broadcast all six date_dim/store inputs, and produce the same shuffle volume (~150 MB), so the difference is not from partitioning, broadcast strategy, shuffle, spill, or GC.

The scans materialize very different amounts of data: each Gluten scan reads 4.1 GB / 1.69B records vs each Comet scan reads 18.9 GB / 8.25B records — the same store_sales table read by each engine. With 4.9× fewer records and 4.6× fewer bytes leaving the Parquet readers, Gluten's store_sales scan stages finish in ~5s of median task time at p50 40–56ms, while Comet's same-shape stages run at p50 299–344ms. The single non-broadcast join is also realized differently (CometSortMergeJoin with two CometSort feeders vs ShuffledHashJoinExecTransformer with no sort), which removes Comet's sort step ahead of the join, but the dominant cost difference is upstream at the scans where the input size delta of 29.6 GB lives.
