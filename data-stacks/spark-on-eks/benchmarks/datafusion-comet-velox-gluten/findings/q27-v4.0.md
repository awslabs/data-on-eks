#### Plan Differences
Same 5 table scans (store_sales, item, date_dim, store, customer_demographics), same 4 broadcast hash joins, same Expand for grouping sets, same TakeOrderedAndProject(limit). Comet plan: 38 nodes / 37 edges. Gluten plan: 54 nodes / 48 edges (+16 / +11).

Operator replacements:
- Scans: 5× CometNativeScan → 5× FileSourceScanExecTransformer.
- Filters: 5× CometFilter → 5× FilterExecTransformer.
- Projects: 6× CometProject → 10× ProjectExecTransformer (+4 extra projects).
- Broadcast join chain: 4× CometBroadcastHashJoin + 4× CometBroadcastExchange → 4× BroadcastHashJoinExecTransformer + 4× ColumnarBroadcastExchange.
- Repartition exchanges: 2× CometExchange → 2× ColumnarExchange.
- Aggregate: 1× CometHashAggregate (single node) → split into FlushableHashAggregateExecTransformer (partial) + RegularHashAggregateExecTransformer (final), introducing one extra shuffle/stage between partial and final aggregate.
- Expand: 1× CometExpand → 1× ExpandExecTransformer.
- Top-N: 1× TakeOrderedAndProject + 1× HashAggregate + 1× WholeStageCodegen → 1× TakeOrderedAndProjectExecTransformer.
- Subquery broadcast (DPP on date_dim): 1× CometSubqueryBroadcast → 1× ColumnarSubqueryBroadcast.
- Columnar/row glue: 1× ColumnarToRow → 1× VeloxColumnarToRow + 2× VeloxResizeBatches + 5× InputIteratorTransformer.
- Code-gen wrappers: 1× WholeStageCodegen (Spark) → 6× WholeStageCodegenTransformer.

Net structural effect: +1 stage (10 vs 9) due to the partial→final aggregate split; same task count (1388).

Broadcast-build stages are negligible in both (1–9 tasks, <300ms each).

#### Root Cause
Comet's 1.78× lead is concentrated almost entirely in the main 631-task probe-side stage. Comet and Gluten run the same logical pipeline over the same store_sales partitions (identical 1,632,143,161 input records, identical ~1.667 M shuffle output rows after partial aggregation), but Gluten's per-task wall time is ~2× Comet's (p50 1.398s vs 712ms; max 2.802s vs 1.172s). The +664 MB input and +25.7 MB shuffle deltas (~3 % and ~16 %) are far too small to explain a 1.96× slowdown. There is no spill, no GC pressure, and no skew change (max/p50 ratios are similar: 1.65× vs 2.00×).

Plan-shape changes that contribute beyond raw per-task throughput: (1) Gluten splits the single HashAggregate into Flushable (partial) + Regular (final) across an added shuffle, producing one extra stage (10 vs 9); (2) Gluten inserts 4 additional ProjectExecTransformer nodes plus VeloxResizeBatches/InputIteratorTransformer/VeloxColumnarToRow glue around the BHJ pipeline. Stage 11396 carries a 224 MB peak executor (off-heap) memory footprint that Comet does not report, consistent with Velox's columnar batch buffering through the join chain.

In numbers: stage 11396 alone is +4.013s slower than Comet's 11435, and the whole query delta is +4.061s — Comet's lead is essentially "the probe-side scan + 4 BHJs + expand + partial-aggregate pipeline is 1.96× slower per task on Velox" on the same data, with the partial/final aggregate split and extra columnar projections/resize-batches as the structural differences accompanying that slowdown.
