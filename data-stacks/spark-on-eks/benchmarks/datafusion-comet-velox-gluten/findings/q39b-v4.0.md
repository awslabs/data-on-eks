#### Plan Differences
57 nodes / 60 edges in Comet vs 88 / 81 in Gluten Velox. Identical 4-table set (`inventory ×2`, `date_dim ×2`, `item`, `warehouse`) joined via 7 broadcast hash joins, 2 partial + 2 final hash aggregates, and 1 sort.

Operator-by-operator replacement (counts equal unless noted):
- `CometNativeScan` (6) → `FileSourceScanExecTransformer` (6)
- `CometFilter` (8) → `FilterExecTransformer` (8)
- `CometProject` (10) → `ProjectExecTransformer` (14, +4 from explicit projections inserted around joins)
- `CometBroadcastHashJoin` (7) → `BroadcastHashJoinExecTransformer` (7)
- `CometBroadcastExchange` (5) + `CometSubqueryBroadcast` (2) → `ColumnarBroadcastExchange` (5) + `ColumnarSubqueryBroadcast` (2)
- `CometExchange` (5) → `ColumnarExchange` (5)
- `CometHashAggregate` (4) → `FlushableHashAggregateExecTransformer` (2) + `RegularHashAggregateExecTransformer` (2)
- `CometSort` (1) → `SortExecTransformer` (1)
- `CometNativeColumnarToRow` (1) → `VeloxColumnarToRow` (1)
- Gluten adds 11 `InputIteratorTransformer`, 5 `VeloxResizeBatches`, 10 `WholeStageCodegenTransformer` wrappers, +1 `AQEShuffleRead`. These are wrapping / batch-management nodes inserted by Gluten's transformer pipeline; they explain the 31-node / 21-edge structural increase but no new computation.

The query shape, join order, broadcast/shuffle topology, and partial-vs-final aggregation split are identical.

Pattern: in both engines exactly one of 20–25 tasks per scan stage processes the full 20 MB of inventory rows (the others get pruned-empty partitions, p50 input 0 B); that single task encapsulates the scan + 4 chained broadcast-hash joins (date_dim, item, warehouse) + filter + partial hash aggregate + shuffle write. Comet runs that pipeline in ~4.5s, Gluten Velox in ~1.6s — the same tail task is ~2.9–3.0s faster, which accounts for the entire wall-clock delta (-3.339s).

Subsequent stages are negligible and similarly faster on Gluten:
- Final agg shuffle stage: Comet 1447/1449 = 480 ms / 623 ms; Gluten 11751/11753 = 272 ms / 297 ms.
- Top-of-plan collect stages: Comet 1452/1458/1462 = 239 + 114 + 16 ms; Gluten 11756/11759/11762/11766 = 86 + 164 + 171 + 20 ms.

#### Root Cause
The query reduces to one CPU-bound task per scan stage that reads 20 MB of inventory and pushes it through a 4-join broadcast chain and a partial hash aggregate before shuffle. Plans are operator-equivalent (same 7 broadcast joins, same partial/final agg split, same 1 sort), so the speedup is execution-time, not plan-shape. On these tail tasks Comet's native columnar pipeline (CometNativeScan → CometFilter → CometBroadcastHashJoin × 4 → CometHashAggregate → CometExchange) takes 4.519s and 4.614s, while Gluten Velox's equivalent (FileSourceScanExecTransformer → Filter → BroadcastHashJoin × 4 → FlushableHashAggregate → ColumnarExchange under WholeStageCodegenTransformer) takes 1.594s and 1.587s — a 65–66% per-task reduction that maps 1:1 onto the -3.339s wall-clock delta. A secondary contributor is shuffle output volume: Gluten emits 24% less shuffle write per scan stage (45.6/45.7 MB vs 59.9/59.9 MB peak; total shuffle write 419.2 MB vs 565.6 MB), reducing downstream shuffle-read work (428.4 MB vs 540.5 MB). No spill, no GC, and equal input bytes (183.5 vs 181.7 MB) on either side, so the gap is pure per-task compute throughput on the broadcast-join + partial-aggregate pipeline executed by the single non-empty inventory partition.
