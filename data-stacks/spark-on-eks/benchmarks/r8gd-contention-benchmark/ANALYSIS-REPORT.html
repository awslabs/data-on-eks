Analysis: Why Similar Performance Despite Higher Stalls?

Why Test 2 Performed Well Despite Higher CPU Stalls

  • Data Locality Benefits: 3 executors on same node can exchange shuffle data locally via memory, reducing network traffic
  • Memory Bandwidth Advantage: r8gd.24xlarge has 768GB RAM with higher aggregate memory bandwidth, less disk spill
  • Transient Nature: PSI spikes (40-50%) are brief events during shuffle phases; average stall remains low (0.68%)
  • Workload Characteristics: TPC-DS 1TB with 768 cores = only ~1.3 GB/core, may not be enough to stress the memory subsystem

When CPU Stalls WOULD Cause Problems

  • Larger shuffle volume: More data = more sustained contention
  • Small shuffle files: Many small files = high syscall overhead
  • Longer-running jobs: Stalls accumulate over time
  • Skewed data: Hot spots become more pronounced

Key Findings

  • No NUMA Contention: r8gd.24xlarge is a single NUMA node (numa_miss=0, numa_other=0). Cross-NUMA memory access is NOT a factor.
  • Transient CPU Stalls: High max stalls (77-97%) are brief spikes during shuffle phases. Average stall is only 1.1-1.6%.
  • L3 Cache Contention: With 3 JVMs sharing 64MB L3 cache on Graviton3, cache misses during concurrent shuffle operations cause transient stalls.
  • No Memory Pressure: Memory pressure is <0.1% across all tests. cgroupsv2 memory.high is not causing issues.
  • No IO Bottleneck: NVMe RAID0 provides sufficient IO bandwidth. IO pressure is ~0%.
